CIM

100w行级别

RuBAC: Rule-Based Access Control 基于规则
RoBAC: Role-Based Access Control 基于角色

业务场景与功能

场景：入职发放多个系统账号，调动调整账号，离职回收账号
目标：只操作一个系统，降低操作成本

  • 多系统连接
    • 通用：文件，JDBC，Restful
    • 定制：SDK，脚本
  • 权限控制
    • 基于规则
    • 基于申请审批
  • 报表审计
  • 元数据维护
    • 自身作为元数据中心
    • 外部导入元数据Data link

架构

核心表

  • 人员表
  • 系统设定表
  • 操作表
  • 账户属性表
  • 执行表

系统设定包括：

  • 映射：人员到账户属性生成规则，外部系统属性和本地账户属性映射，即每个外部系统属性关联一个本地账户属性(UserData)以及人员生成规则
  • 范围：系统人员规则，系统每种权限规则，比如所有开发人员都有Gitlab账户，不同组的开发对应不同项目的成员

操作流程：即修改人属性，执行操作计算影响到哪些系统和系统内哪些权限，尝试用本地账户

  1. 修改人
  2. 产生操作
  3. 执行操作
  4. 根据生成规则产生本地账户以及对应执行数据
  5. 执行数据根据系统映射产出外部系统属性

执行操作事件是典型的事务行为，即产生多个本地账户和执行记录，在分库分表环境下可以按照人员进行切分，即同一人的本地账户和执行记录在同一机器上，依然可以使用本地事务

设计细节

请求执行器无状态设计，无设施依赖

  • 增加了执行器之间协调的难度
  • 减少了维护成本

系统连接信息本地缓存，CIM提供API

  • 不用随每次发送，减少发送量
  • 执行器按需请求

同一事件由多个系统执行，不是分布式事务关系，其中一个失败不会导致其他回滚，尽最大努力重试模型

导入流程

  1. 导入人员元数据Person
  2. 目标系统导入，元数据账户数据匹配关联,建立UserData
  3. 修改元数据，根据设定生成新版本UserData
  4. 执行Apply发送给目标系统

两级执行机制

如果改动元数据直接修改目标，过于分散无法管理

  • 事件层能集中管理变动，可以撤回
  • 事件层能进行合并，比如同一天多次mod和并，有效化和新增合并

两级执行

  1. 改变元数据
  2. 产生事件
  3. 执行事件，本地账户变化
  4. 产生各系统请求
  5. 执行请求，远程目标系统账户变化

设计模式

  • 单例模式，通过Spring实现
  • 迭代器模式，分页结果返回，一页消费完处理下一页，微软Graph API
  • 模板方法模式，导入逻辑包含数据导入，数据匹配，数据检查，公共导入逻辑实现框架，具体连接器
  • 责任链模式，通过Filter实现，某些请求地址上需要添加新建窗口请求参数，后端调整
  • 命令模式，构造好后交给线程池处理

分库

每个用户一个库

未来改进

  • 数据表分离，当前数据和历史数据，每次修改时，利用当前旧数据插入历史数据，同时修改当前
  • 库融合，多租户方案

属性体系

属性绑定

描述账户属性
CIM自身数据库维护一套
目标系统维护一套

属性填充

加入人需要产生账户属性

  • 来源于人属性
  • 固定值
  • 生成器(能力最强，包括上述两种)
    • 随机文字
    • 自增
    • 时间
    • 所属部门属性

自维护

本身也作为一个导入系统，区分用户权限

普通用户功能

  • 申请账户/权限
  • 修改密码
  • 人员检索
  • 组织架构图

服务端提供API

  • 获取连接信息
  • 获取token
  • 提交执行结果 apply
  • 提交导入结果集 inventory
  • 获取账户系统列表

定期清点

审计，在某一时刻，所有账户的状态

规则：

  • 相符，应该存在账户
  • 不符，不应该存在
  • 无人，没找到匹配

账户

  • 存在，账户存在
  • 缺失，物理不存在

状态

  • 有效
  • 无效，可以认为逻辑不存在

流程

  1. 读取目标账户
  2. 尝试匹配CIM账户
  3. 匹配到CIM账户，通过账户获取人
  4. 没匹配到CIM账户，尝试直接匹配人
  5. 剩余CIM没匹配的账户，可以获取人

约束
无人时CIM侧账户只能缺失

人符合规则 CIM侧账户信息 状态 目标侧账户信息 状态 账户属性一致 结果
符合 存在 存在 一致 COMPLETE_TYPE 正常状态
符合 存在 存在 差别 RULE_WIN_MOD_TYPE_NOP 属性不一致
符合 存在 缺失 RULE_WIN_ADD_TYPE_NOP 目标缺失
符合 缺失 / UNMATCHED_TYPE_CIM_GAP 违反CIM规则 应该有账户，但是CIM没有
不符 存在 有效 / UNMATCHED_TYPE_CIM_GAP 违反CIM规则 不应该有账户，但是CIM有
不符 存在 无效 存在 有效 UNMATCHED_TYPE_DIFFERENT_RESOURCE 不应该有账户 目标有账户
不符 存在 无效 存在 无效 BOTH_INVALID_TYPE_NOP 可以认为正常，有所区别
不符 缺失 存在 DEL_TYPE_NOP 目标不应该出现的账户
不符 缺失 缺失 COMPLETE_TYPE 正常状态
没人 缺失 存在 NONE_OWNER_TYPE_NOP 目标无人认领不受控账户

连接器

基本API功能

  • 获取全用户列表 GetAllAccount
  • 增删改用户
  • 有/无效化用户
  • 获取权限类别所有可选项 GetAllRole
  • 获取特定权限下成员 GetRoleMember
  • 增删权限项

Restful连接器

  • 自定义请求状态检查
    • 验证不成功应该401，返回200并且页面包含错误信息
  • 请求体构造
    • Json可视构造，解析路径表达式JsonPath(aaa.bbb[1].ccc)
    • form表单
    • text纯文本
  • 分页功能(指定页数/偏移量/下一页链接)
  • 写回功能
    • UserId写回，新增用户，id由目标系统生成
    • RolememberId写回，部分系统把成员当成资源，删除需提供id
  • 合并请求，需要多API
    • 级联请求(有先后顺序，第一个请求返回信息值，作为第二个请求参数)
    • 并行请求(无先后顺序，单一请求信息不够，需要从多个端点汇聚)
  • 批量特殊处理
    • Json Batch打包成一个
  • 限流处理
    • Redis维护每种请求类型下一次尝试时间
    • Redis类似令牌桶发送端主动控制请求频率，不精确但是比滑动窗口省空间。比如控制每分钟100次，极限可能200次，比如很久之间更新一直没用，突然全访问完并且由于间隔时间久又补了100，瞬时可能200
  • 模板化
    • SCIM
    • GitLab

token过期处理

消费者内存维护当前token，生产者Redis维护旧新token映射表以及最新token

  • 同步token流程
    消费者自身创建队列，订阅fan-out广播交换器
  1. 消费者启动时自身没有token
  2. 调用API获取token
  • 申请新token流程
  1. 很多消费者发现token过期
  2. 调用API发起申请
  3. 服务端分布式锁，只有一个通过其他拒绝，通过的消费者申请新token，其他消费者等待
  4. 申请到新token后，调用API发给服务端
  5. 服务端存储新token
  6. 服务端下发广播消息，所有相关消费者收到

批处理时顺序保障

有依赖的专门打包成大消息执行

消息重复处理

RabbitMQ消息重复

  • 增加账户时，利用目标系统唯一约束，多次操作直接失败
  • 同一用户期间内多次修改预先合并，短期内只有一个修改，重复执行无影响
    如果重复执行服务端会受到重复应答，如果发现已经成功又有应答，启动检查机制，比对目标系统当期值是否符合预期

熔断降级

Hystrix线程池隔离 请求多目标系统 防止单系统崩溃造成资源全部占用导致整体停滞
降级Semaphore请求Redis，不行走内存Session
