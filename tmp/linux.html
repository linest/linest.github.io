linux

内核有资源CPU、存储、IO, 系统调用是内核提供的接口，用户不能直接访问内核资源只能直接或间接使用系统调用
库函数是编程接口，便于编程，实现上封装了系统调用，屏蔽了复杂性，并且加入缓冲提升性能。很多库函数和系统调用同名，比如read write

用户态到内核态切换方式

  • 系统调用：主动请求
  • 异常：比如缺页
  • 外围设备中断：设备完成用户请求的操作后比如硬盘读写，向CPU发中断，CPU会暂停执行下一条指令转而去执行与中断信号对应的处理程序

文件句柄/描述符

文件描述符是一个非负整数，内核为了高效管理已经打开的文件，创建索引
默认1024，即单个进程最多只能接受1024个连接

进程线程

内核不区分进程线程，普通进程和轻量级进程
进程由fork创建， 线程由pthread_create创建，两者最终都调用do_fork

do_fork有很多参数，CLONE_VM（共享内存空间）、CLONE_FS（共享文件系统信息）、 CLONE_FILES（共享文件描述符表）、CLONE_SIGHAND（共享信号句柄表）和CLONE_PID（共享进程ID，仅对核内进程，即0号进程有效）即轻量级进程共享进程资源

调用fork后，创建一个新的进程控制块(PCB)，子进程完全拷贝父进程地址空间(堆，栈，代码段等)
CopyOnWrite, 一开始只读共享，如果写操作会触发缺页异常，让父子进程拥有各自的内存页

在fork函数执行完毕后，如果创建新进程成功，则出现两个进程，一个是子进程，一个是父进程。在子进程中，fork函数返回0，在父进程中，fork返回新创建子进程的进程ID。可以通过fork返回的值来判断当前进程是子进程还是父进程。

父子进程执行同样的逻辑，如果子进程想执行其他，可以exec, 删除pcb信息从磁盘调入新代码，形成新的pcb

线程模型

  • 一对一：一个用户线程对应一个内核线程， 内核负责调度，Linux2.6 NPTL
    • 优点：实现简单
    • 缺点：用户内核态切换频繁，实体对应数量大时影响性能
  • 多对一：多个用户线程对应一个内核线程，调度等由用户空间线程库处理
    • 优点：用户态很多操作对内核透明，无需切换
    • 缺点：内核不知道有哪些线程无法完整调度，用户线程阻塞会影响其他用户线程
  • 多对多：实现复杂

线程切换

场景

  • 时间片轮转，正常用完
  • 线程阻塞，进行不下去
  • 资源没满足，进行不下去
  • 主动放弃

开销

  • 用户态内核态切换
  • 上下文切换，(寄存器、计数器、线程栈)
  • 缓存缺失

进程间通信

  • 管道
    文件作为中间介质。单向通信，写进程和读进程。可以局限亲缘关系进程临时的无名管道(pipe)和永久的命名管道(fifo), 传输速度慢
  • 消息队列
    消息缓冲区为中间介质，消息的链接表，存放在内核中
  • 信号量
    不能传递数据，只用于同步
  • 信号
    比如Ctrl+C发出SIGINT，关闭终端session发出SIGHUP
  • 共享内存
    开辟一块内存，映射到各个进程的虚拟地址空间，修改对各进程都可见
    优点是效率很高避免了数据内核/用户间拷贝，缺点是没有同步机制容易混乱
  • Socket
    可以用于不同机器间

虚拟内存

虚拟内存为进程提供私有独立逻辑地址空间，有隔离作用，进程不直接操作内存而是操作虚拟内存，操作系统完成虚拟内存到真正物理内存的映射，避免了多个进程同时操作真实地址空间
不连续的物理内存可以在虚拟内存层面连续
虚拟内存可以大于真正内存，暂时不用的数据会映射到磁盘预留的Swap分区，需要时调入内存
虚拟内存还可以设置共享页，实现多进程内存共享

虚拟内存结构

  • 内核代码和数据
  • 栈空间，向下增长
  • 堆空间，向上增长
  • 用户代码和数据

内存管理单元MMU完成映射

内存映射

通过虚拟内存访问，如果双方虚拟内存地址映射到同一段物理内存地址，那么实际上操控同一段实现了共享

磁盘文件读取

ext文件系统中，对一个文件的读需要先从磁盘中读取到文件对应的Inode，然后根据inode获取到对应的block的位置信息，进行第二次磁盘读取目标文件的内容，这样就需要两次磁盘IO。在系统中会有inode缓存，这样可保证一些文件读取只需要一次磁盘IO

小文件存储系统通过小文件合并，减少文件数量，尽可能多的INode保存在缓存，提升效率

多路复用

监视多个描述符fd，就绪就通知进行操作
select，poll,epoll都是同步IO, 就绪后自行读写，而异步IO是拷贝到用户空间，直接使用

select 和 poll

遍历描述符列表，检查是否有就绪

POSIX标准
知道有事件发生，不明确哪一路，轮询遍历复杂度O(n)

将描述符从用户空间拷贝到内核空间中，遍历描述符，调用poll方法检查，遍历完没发现有就绪则挂起，直到设备就绪或者超时候，被唤醒再次遍历

描述fd集合的数据结构有所差别

  • select采用的结构有数量限制32位默认1024，64位默认2048
  • poll采用的结构基于链表存储，没有最大连接数限限制

缺点

  • 每次都需要拷贝数据到内核
  • 数量大时遍历时间变长

epoll

描述符就绪后通过回调函数加入就绪列表，只遍历就绪列表

Linux特有
事件驱动，事件和描述符关联，复杂度O(1)
一次性拷被描述符
每个描述符关联回调函数，就绪后执行回调加入到就绪链表，只需要不断遍历就绪链表检查是否为空即可

连接数有上限但是很大，1G内存能打开10万连接
不通过遍历，性能不会随着数量增多而下降，而且只有活跃连接才触发事件，和总连接无关
