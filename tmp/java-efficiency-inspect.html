内存监控

堆溢出自动保存dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=C:\temp

FullGC时也可保存
-XX:+HeapDumpBeforeFullGC
-XX:+HeapDumpAfterFullGC

打印GC日志
-XX:+PrintGC
-XX:+PrintGCDetails
-XX:+PrintGCTimeStamps
-XX:+PrintGCDateStamps
-Xloggc:C:\log

允许JMX远程连接
-Dcom.sun.management.jmxremote
-Dcom.sun.management.jmxremote.port=9010
-Dcom.sun.management.jmxremote.local.only=false
-Dcom.sun.management.jmxremote.authenticate=false
-Dcom.sun.management.jmxremote.ssl=false

Runtime.getRuntime().freeMemory() 当前已获取内存下的空余内存
Runtime.getRuntime().totalMemory() 当前已从系统获取到的内存
Runtime.getRuntime().maxMemory() 受xmx限制，能从操作系统获取内存的上限

MAT
Eclipse Memory Analyser Tool
http://www.eclipse.org/mat/downloads.php

jmap

jmap查看堆情况

展示各class的实例数，内存占用情况

先gc一次再显示，手动触发了一次full gc

导出dump，也会先gc

jstat

显示gc相关数据

  • C容量 U使用量
  • S0S1幸存区 E伊甸区 O老年区 M元数据
  • YGC新生代GC次数 FGC全GC次数 YGCT/FGCT所用时间秒数

jps

显示进程，同时显示详情，可以看到启动路径(l),相关JVM参数(v),main方法参数(m)

jinfo

输出全部配置

输出所有指定的非默认JVM参数

jstack

查看进程下的线程栈,可以看到死锁状况

统计线程状态

visualVM

jvisualvm, 常见命令的界面化综合工具

MAT 分析
http://www.importnew.com/7869.html

Dump文件过大

如果文件大小大于本地内存大小，那么可能分析工具OOM
需要配置分析工具的JVM参数,比如8g内存去分析16g的文件

CPU监控

JProfile
Yourkit
jstack采样，占用高的方法出现频率高

服务器CPU达到95%，瓶颈

Linux系统性能排查

负载

  1. uptime查看负载
  2. top找到占CPU高的进程ID
  3. top -Hp 进程ID定位进程下具体线程
  4. printf %x 线程ID转成16进制
  5. jstack 进程ID | grep -A 200 线程ID查找线程栈，显示200行
    也可以先重定向到文件，然后文件查找
    https://fastthread.io/ 可以可视化

内存

  1. free -m 以MB为单位列出内存情况

进程和线程

  1. ps -mp 进程号 -o THREAD,tid,time

数据库

  1. show full processlist 查看执行语句
  2. explain 查看执行计划
  3. 开启慢查询

tomcat文件数

  1. ps -ef | grep tomcat 找到线程id
  2. lsof -p 线程号 | wc -l 统计打开文件数量
  3. ulimit -a 查看允许上限

定时任务

  1. crontab -l查看定时任务
