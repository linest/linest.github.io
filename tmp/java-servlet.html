Java Servlet框架

Server Applet，提供编写服务端程序的框架

servlet-api提供了编程接口，实际上是Servlet容器服务器提供了实现，比如Tomcat,Jetty
Tomcat 7 默认使用Servlet3.0
Tomcat 8 默认使用Servlet3.1

每个应用处在一个独立的Servlet容器

Servlet

javax.servlet.*提供通用接口
Servlet接口主要定义了生命周期方法init-service-destroy
Servlet只会被初始化一次，默认是延迟初始化，收到第一个请求时触发，也可以通过配置在应用启动时就初始化
Servlet线程间共享，避免定义成员变量造成多线程共享问题
容器关闭或者容器决定关闭(长时间没用)，会执行销毁方法

Servlet有个抽象类GenericServlet，不涉及协议，主要提供了一些简化方法

  • 直接实现ServletConfig，可直接使用配置相关方法
  • 提供无参版本init方法

ServletConfig与配置文件

这些配置在运行时由ServletConfig维护

ServletContext

ServletContext是全局应用级别，应用部署时由容器创建产生，一个web应用只有一个
ServletContext包含在ServletConfig中，传递给Servlet，每个Servlet专属
GenericServlet实现了ServletConfig接口实现了getServletContext()，即每个Servlet都可以直接拿到ServletContext
ServletContext不仅仅可以获取当前应用，还可以用getContext()方法传入路径可以获取同服务器下其他web应用的上下文

RequestDispatcher

资源的包装类，资源可以是servlet，文件等等

RequestDispatcher可以实现Servlet Chaining，即多个Servlet共同处理，同应用跨应用都可以
处理过程就是对请求ServletRequest和响应ServletResponse这两个对象不断加工
RequestDispatcher可以从ServletRequest获得，也可以从ServletContext获得，区别是从ServletRequest可以使用相对于当前Servlet的路径

Filter

Filter和Servlet类似，都是对请求和响应进行处理，更多在角色上有差异
Servlet是最终业务逻辑，Filter进行预处理和后处理，请求先通过Filter链到达Servlet响应再反向通过Filter
即一个Filter会经过两次，以用户调用doFilter时为分界，之前的代码是预处理逻辑，之后的代码是回来时的后处理逻辑
Filter接口上也和Servlet相似，不同点在于Filter有个链式结构，在设计上就是考虑多个Filter链式执行
Filter在应用启动时就创建

链的作用就是转到下一个Filter处理

和ServletConfig功能类似，有一个同样的FilterConfig

Listener

  • 监听目标：请求，上下文
  • 监听行为：建立销毁，数据增删改

在servlet或者filter初始化之前通知

应用端配置

Servlet也是一种资源，只是一个处理类，代码实现时并没有涉及路径绑定，Servlet初始化接受一个参数，代码实现也没有涉及，这些都是通过配置文件完成
web.xml是应用端配置，相当于提供注册功能

  • 定义应用包含哪些Servlet，已经加载行为
  • 定义Servlet和路径的映射关系
  • 定义参数，应用上下文级参数和Servlet级参数

Servlet3.0以后，取代了web.xml的功能

  1. ServletContext可以用代码进行添加, 不再像以前一样只能来源配置文件
  1. 引入注解java.servlet.annotation.*
    @WebServlet,@WebFilter,@WebListener各组件使用注解自我描述配置
  1. 执行配置逻辑
    ServletContainerInitializer可以在启动时执行特定逻辑
    用户提供实现类，并在实现类上使用@HandlesTypes注解标记感兴趣类型，这样启动时这些类型会作为Set传入，同时能拿到ServletContext，可以进行配置
    META-INF/services目录创建一个名为javax.servlet.ServletContainerInitializer的文件，内部指明实现类，这样容器启动时就会调用

服务端配置

server.xml是应用服务器端的配置
ServletContext来源于容器，除了包含应用端配置，还包含服务端配置

  • <Context>标签下<Parameter>可以配置上下文变量
  • <Logger>标签可以配置日志
  • <Context>的crossContext要设为true才能允许跨应用访问

HTTP相关实现

javax.servlet.http.*在通用基础上提供HTTP接口

HttpServlet

HttpServlet是GenericServlet的子类，依然是抽象的，限定了HTTP协议，提供HTTP相关操作方法
HttpServlet响应不同的请求可选类型GET,POST,PUT,DELETE,OPTIONS,TRACE
Servlet3.1里并没有PATCH，必要时可以覆写service方法添加支持
HttpServlet提供的所有doXXX响应方法都是空壳实现，用户编写的子类需要覆写方法来表示支持

HttpServlet实现了一套处理流程，这样用户实现时只需要提供doXXX方法，无需修改service方法
用户实现可以直接继承GenericServlet，实现service方法，完全把控所有处理流程

Cookie

描述了单条key-value信息

Cookie供浏览器存储，随响应发回，随请求发出

HttpSession

HttpSession就是服务端维护的一组key-value，sessionId给客户端作为识别

HttpSession只跟请求有关，默认行为是没有就新建，可以传参控制

HTTP无状态，肯定是请求中给了sessionId，服务端才能对应
通过请求可以获取sessionId并判断来源及有效性

3.1版本支持更换SessionId

Http相关Listener

Http专属监听器围绕着HttpSession，建立销毁，数据增删改

3.1版本id可变，新增加一个监听器

除了上面面向容器管理的监听器外，还提供两个面向数据的监听器
这两个接口由数据类实现，在数据对象加入或者退出，遇到钝化与激活时执行特定行为
这种监听器由程序逻辑控制无需手动注册
