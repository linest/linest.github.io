TCP&UDP

IP层指定TCP还是UDP
IP地址+端口+协议类型作为唯一标识，即TCP和UDP可以同时指定同一端口号

UDP

User datagram protocol
简单不可靠传送，没有TCP的复杂机制
基于数据报文，非流式传输，保留应用层数据边界

UDP报文格式

适合场景

  • 网络情况好的内网，丢包不敏感
  • 广播
  • 处理速度快，低延时
    流媒体协议，直播
    实时游戏
    物联网

应用协议

  • DHCP
  • RIP
  • DNS

UDP没有拥塞控制，但也不能无节制发包，毕竟资源有限
可以搭配其他协议，基本原理是接收端应答一些（丢包，延时等）统计信息，供发送方调整速率

  • DCCP(Datagram Congestion Control Protocol) 数据报拥塞控制协议
  • RTP(Real-time Transport Protocol) 实时传输协议

TCP

Transmission control protocol
面向连接的传输层协议，可靠传输
四元组唯一确定：源IP地址，源端口，目的IP地址，目的端口

因为建立了连接，可以基于字节流交互，有状态服务

TCP报文格式

应用协议

  • HTTP
  • FTP
  • SMTP
  • TELNET

可靠性

  • 无差错: CRC校验，接收时先检查正确性
  • 不丢失：确认应答，超时重传
  • 不重复：唯一序号，重复接受无效
  • 按序到达： 缓存，无序变有序

应答与重发机制

既然是可靠传输，就要保证对方真的收到
对方已经收到的证据是对方发送了应答

应答

并不是确认一个包收到后再发下一个，而是一次发很多，发送方和接收方都维护缓存
为了区分同时发出的数据包，每个包关联一个序号
发送方：发送已确认，发送未确认，等待发送，暂时不发送，中间两部分是发送窗口，大小由接收方提前指定
接收方：已接收，待接收，待接收部分构成接收窗口

延迟应答
收到后不会立即应答，而是固定周期发起

累计确认
应答是所需要的下一个序号，即当应答一个序号时，表示前面的都已经收到

延迟和累计好处是合并了多个响应包减少开销，甚至可以在接收方发送数据时捎带确认

重发

如果发送方迟迟没有收到应答，就认为没收到，重发

动态超时时间
网络的环境是不同的，有快有慢，时好时坏

  • 如果网很快，重发超时时间太长，丢包就不能及时感知
  • 如果网很慢，重发超时时间太短，可能有多余的重发，浪费网络资源

RTO(Retransmission TimeOut)重传时间
RTT(Round-Trip Time)往返时间

所以超时值RTO不能固定，需要动态测量,自适应超时算法，采样RTT

  • 每次都会记录RTT时间，并计算方差，超时时间根据这些数据确定
  • 被重发的数据再次超时，那么超时时间以2倍的方式指数增加
  • 超时一般是0.5秒的倍数，初始时没有统计数据，默认6s

快速重传
超时机制属于被动机制，发送方要等
还有主动机制，发送方收到三个冗余应答，表示丢包概率很高，立刻重发
接收方收到乱序包，立即发送冗余应答，比如1,2号包收到，3没收到，收到4，那么就应答3，表示想要3号包
这一机制下，不丢包的乱序会导致40%概率收到3次应答，而丢包会100%收到，因此确定三次

重传歧义
对于数据而言，序号是固定的，重发也是一样的序号
如果超时重发后，收到响应，无法确定是原来数据响应还是重传数据响应，影响RTO计算

流量拥塞控制

流量控制

滑动窗口，防止把接收方缓存塞满
接收窗口(rwnd)不是固定的，比如应用一直不读取接收的数据，没有空间缓存后续内容
接收方每次应答时，还发送窗口大小，这样发送方相应调整窗口，甚至停止发送
发送未确认窗口大小 + 待发送窗口大小 = 接收窗口

拥塞控制

网络容量=带宽*往返延迟，同时传输的数量有限
如果没控制，网络变慢超时后都重发，那么网络上负担更重，恶性循环
拥塞窗口(cwnd)，防止把网络塞满，主动减少发送
发送未确认部分即能发的部分同时受流控和拥塞制约，即发送未确认部分窗口大小 = 接收窗口和拥塞窗口的最小值，如果接口窗口小说明接收方制约，如果拥塞窗口小说明网络环境制约
发送方无法直接检测外界，只能从侧面感知拥塞，发送太多超出网络处理速度，要么被中间设备丢弃，要么被中间设备缓存导致超时，重发就是感知的标准
慢启动

  • 从只发1个开始，每次收到全部应答后发送数量二倍指数增长，涨到65535(ssthresh)时改成线性增长(拥塞避免)
  • 普通重发算法，遇到重发，窗口重置为1，ssthresh变为cwnd窗口一半
  • 快速重发算法，遇到连续三次冗余应答，窗口减至一半左右，ssthresh变为cwnd窗口一半，线性增加

机制问题

  • 丢包不一定是发太多，可能网络本身就会丢
  • 塞满缓存才能感知，时机太晚
    算法折中权衡，比如BBR

建立连接

全双工协议
三次握手，客户端主动

LISTENING：服务端监听状态，可接受连接
SYN_SENT： 客户端发起SYN后
SYN_RCVD： 服务端收到SYN后
ESTABLISHED： 连接建立
-> SYN=1, seq=x
<- SYN=1, ACK=1, seq=y, ack=x+1
-> ACK=1, seq=x+1, ack=y+1

因为涉及到重传，序号的作用是区分具体是哪一次的数据
发起时序号由计数器产生，每4微妙增加1，这样序号循环要4个多小时，序号主要可以区分同一端口上的请求，避免序号冲突混乱

三次握手实质上让双方验证自身的收发能力和对方的收发能力，信息对等

  • 客户端收到应答，说明服务端发送没问题，服务端接收没问题
  • 服务端收到应答，说明客户端发送没问题，客户端接收没问题

客户端收到就进入连接状态，而服务端收到需要再次二次确认
这样能避免超时到达而建立脏连接，客户端请求可能延迟，可能重发请求，如果服务器第一次应答就建立连接，可能建立脏连接

连接在系统层面实际上是对应文件句柄，在逻辑上是通信两端的状态维护，并不存在一个实体连接，只是占用了一个端口通道

断开连接

四次放手
双方各主动发送一次，告知不再发送，但是可以接收

主动方状态
ESTABLISHED
FIN_WAIT_1 主动方发起FIN后
FIN_WAIT_2 主动方收到ACK后
TIME_WAIT 主动方收到FIN后
CLOSED

被动方状态
ESTABLISHED
CLOSE_WAIT 被动方收到FIN后
LAST_ACK 被动方发起FIN后
CLOSED

-> FIN=1, seq=u
<- ACK=1, seq=v, ack=u+1
<- FIN=1, seq=w
-> ACK=1, seq=u+1, ack=w+1

FIN和ACK可以组合成FIN+ACK一并发送

最后一次发送ACK后，进入TIME_WAIT，等待2MSL（maximum segment lifetime，最大生存时间，MSL建议2分钟）之后才关闭，主要是基于网络不可靠性考虑

  • 防止应答丢失，最后发送可能网络问题没收到，对方会重发，还需要响应
  • 防止混淆新旧连接，太早释放可能马上用来建立新连接，会遇到之前连接上的信息到达

主动关闭一方会进入Time Wait, 如果服务端主动关闭，并发又很大，那么大量连接不能建立

/etc/sysctl.conf控制相关参数,sysctl -p生效
缩短等待时间到30s
net.ipv4.tcp_fin_timeout = 30
开启重用
net.ipv4.tcp_tw_reuse = 1

连接细节

  • Backlog队列
    建立连接过程中，服务端维护队列

    • syns queue： 收到syn，等待响应的半连接
    • accept queue： 已完成握手，但是还没accept的连接
      backlog参数和系统参数somaxconn取小值控制队列大小,超过队列大小连接会被拒绝

  • SYN Flood攻击
    伪造大量不存在的ip发syn包，服务端收到后进入SYN_RECV状态并加入未连接队列
    服务端的响应肯定无人应答，服务端会不断重试很长一段时间才放弃，耗费资源，同时占用队列可能导致正常请求无法连接

  • SYN Cookies
    服务端不维护syn队列,好处是不会丢弃请求，而是进行哈希出一个cookies作为序号，后期建立连接会验证序号合法性

  • 大量CLOSE_WAIT状态
    进入CLOSE_WAIT说明是被动关闭，如果迟迟不发起主动关闭，会一直维持在这个状态，没有超时时间
    通常原因是代码漏洞，没释放资源
    比如数据库主动发起断开连接，服务端收到后进入CLOSE_WAIT，但是服务端代码忘了释放连接，一直持有
    比如业务服务器请求资源服务器，如果资源不存在，那么资源服务器主动发起断开，此时业务服务器代码忘了关闭资源，业务服务器一直处于CLOSE_WAIT状态

还有可能BackLog太大，服务器还没来得及处理，客户端已经超时关闭了

  • 大量TIME_WAIT状态
    主动关闭一方会进入TIME_WAIT状态，并持续2MSL通常4分钟才关闭
    如果发起大量短请求比如爬虫，那么会导致大量TIME_WAIT连接，占用资源影响新连接建立
    可以适当调低参数减少等待

KeepAlive

TCP级心跳，发送空报文

  • 检测连接是否存活
  • 防止网络不活动而断开
    和应用级心跳相比，更轻量，但没有业务逻辑

KeepAlive机制可以清理异常CLOSE_WAIT，但默认两小时才发起一轮检测，可以适当减小

粘包问题

同一连接上连续发送，一个完整消息可能会被拆分成多个，也可能多个包封装成一个进行发送
Nagle算法，发送数据量太小就暂时不发，等数据多了一起发，避免频繁发送小包减少开销
TCP流传输，不区分业务结构，只考虑字节，导致不完整不相关数据在一起发送

解决思路是加入边界信息，便于应用层识别完整的包

  • 数据前先写入长度信息
  • 插入不影响协议的特殊符号

TCP的缺点

性能

  • 慢启动： 初始速度慢
  • 抗抖动差： 网络一旦丢包会降低速度

lsof
