Java ThreadLocal

  • 多线程
  • 源码
Java

每个线程维护单独的数据副本，形式上实现变量的无锁共享

功能上类似于用线程id为key的map,每个线程独立

内部实现

虽然在功能上像个Map，但是实现上完全不同
Map对所有键值对集中管理，而ThreadLocal是调用的线程各自维护,谁用谁负责

  • 所有的操作都是基于Thread.currentThread()，每个线程都有ThreadLocal.ThreadLocalMap threadLocals,用于存储线程用到的变量
  • 变量和线程之间是低耦合的，在运行调用时才建立联系。线程提前不知道会使用多少ThreadLocal,ThreadLocal也不知道会有多少线程

ThreadLocalMap

名字里有map,但根本没实现Map接口，是个专有的特殊结构
底层依然是Entry数组，但是ThreadLocal本身作为key是弱引用实现，即考虑大数据量的自动释放

哈希冲突的解决不是链表，而是开放地址
查找时先看是不是直接命中

没命中的情况下，会顺次向后循环查找
值得注意的是如果发现key值为null,那么说明已经通过弱引用机制回收了，会把Entry清理出去

存值操作

线程的threadLocals初始是null,用到时才建立ThreadLocalMap
ThreadLocal只作为key，并不持有value，value是ThreadLocalMap持有的

取值操作

ThreadLocalMap是以变量本身作为key

初始化操作

ThreadLocal有一个提供初始值的initialValue函数，默认是null,子类可以重写
当线程调用get没找到值时就会调用该函数，包含两种情况

  • 线程第一次调用get并且之前没被set
  • 线程调用remove清空后再调用get

清理操作

一般情况下不用清理，使用结束后JVM清理
但是搭配线程池使用时，由于线程会重用，注意在开始时及时调用remove清除遗留数据
另外如果ThreadLocal被声明静态，那么不会失去引用，弱引用清理的机制不会生效，造成内存泄漏

InheritableThreadLocal

ThreadLocal还有一个子类，可以用于传递给子线程
和普通的ThreadLocal不同的是，在Thread类中保存的位置不同，这个位置的数据会复制给子线程

JDK使用

java.util.concurrent.ThreadLocalRandom
Random本身是线程安全的，ThreadLocal版本可以避免多线程下对随机seed的争抢，提升性能

应用

DateFormat线程不安全,可以用ThreadLocal包装
