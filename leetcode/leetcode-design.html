LeetCode-设计篇

总结

注意

  • 队列和栈弹出操作前考虑空

附加功能

❀155. Min Stack

增加getMin功能，操作复杂度O(1)

  • 双栈版本
    额外维护最小值堆栈，注意等于当前最小值时也要再次入栈，防止最小值提前弹出
  • 单栈版本
    所有操作都同时操作两个数据，最小值和数据作为整体
    额外维护最小值，那么把数据放在上面，方便top操作

❀380. Insert Delete GetRandom O(1)

支持随机获取的集合，要求操作O(1)

支持插入删除需要set即可，但是使用随机需要支持下标访问，采取set配合list
list删除时需要后续移位，并且破坏了下标连续性，采取策略每次删除把值交换到末尾

❀284. Peeking Iterator

支持peek操作的迭代器

表面上peek,实际内部迭代器已经取值，相当于维护一个预留空间

  • null是Integer合法值，并不能用来代表预留值不存在，需要布尔变量配合
  • peek操作如果预留有效直接返回，无效的话取值填充再返回
  • next操作直接使用peek,取值操作导致预留无效化
  • 只有预留无效，并且底层迭代器也没数据，这时彻底没数据

❀341. Flatten Nested List Iterator

能处理多层嵌套数据的迭代器

NestedInteger一个对象对应实际上不确定数量的元素

  • List可能为空，因此即便NestedInteger可能不包含元素
  • 只有真正找到了Integer值才能判定hasNext，遇见List继续分解
  • 使用栈作为临时存储，优先深度，同时逆序押栈，确保头部元素最先使用

❀251. Flatten 2D Vector

展开二维向量，即列表内每个元素是一个列表，并且不等长

注意有空数组，内部维护行列索引，要注意保持永远有效
迭代器hasNext和next都可以先调用，不管哪个先调用都要保证可以正常工作

❀173. Binary Search Tree Iterator

二叉搜索树迭代器

中序遍历，栈维护
迭代器靠hasNext确保有值，next内无需检查

流

❀703. Kth Largest Element in a Stream

维护流数据中第k大的数

使用容量为k的最小堆，不断弹出小的堆内保留大的，维持容量，即维护了全局最大的k个数
注意先插入超出容量再弹出，不能先弹出再插，因为弹出的可能是刚插入的

❀295. Find Median from Data Stream

支持找中值

左半部分大顶堆，可以取左边最大
右半部分小顶堆，可以取左侧最小
新元素可能应该维护在左侧也可能在右侧，先无脑加右侧再弹出，再加左侧再弹出，就是新的受影响改动的值
调整左右容量，保持左小于等于右

特定结构实现

❀232. Implement Queue using Stacks

用栈实现队列

  • 一个栈专门用来压入，另一个用于输出
  • 输出栈为空，尝试将输入栈数据弹出再压入输出栈
    虽然pop操作可能对输入栈进行遍历，但是每个数据只会进入输出栈一次，总体上复杂度还是O(1)

❀225. Implement Stack using Queues

用队列实现栈

双队列方案 push O(1) top pop可能O(n)

  • 一个队列用于操作，一个队列用于临时存储
  • 需要弹出或访问栈顶时，只留操作队列最后一个元素，其他转移
  • 如果操作队列为空，交换两个队列

单队列方案，每次压入元素，滚动到队列头部push O(n)

基础实现

❀707. Design Linked List

实现链表常规功能

维护头节点和size，辅助函数find更加简便

❀208. Implement Trie (Prefix Tree)

实现前缀树

节点不存字符，通过上级节点是否有下属判断

❀146. LRU Cache

值为正数的缓存,替换最老不访问的数据，不存在返回-1

存取操作O(1)

  • 需要维护访问顺序
  • 不能遍历链表需要直接定位机制

方案1，直接使用LinkedHashMap
方案2，LinkedHashSet兼具链表和定位功能

方案3，手动实现链表

  • 普通的list只能获得值而无法获取节点引用，因此自定义结构并将整个node作为map的值
  • 根据某个节点引用移除节点，节点内双向链表结构，便于找前序节点
  • 容量满弹出时，需要根据节点删除对应key，因此节点内需要存储key
  • 维护链表头尾指针，采取伪节点，避免复杂null判断
  • 两个核心函数，链表追加和节点删除
  • put操作注意key已有情况，避免错误的LRU替换

❀460. LFU Cache

值为正数缓存替换最低访问频次的数据，如有多个替换LRU

存取都是O(1)操作
首先要有Map实现基础缓存功能
因为有频次，需要有Map进行频次统计
为了LRU,需要Map维护同频次列表，使用LinkedHashSet保证顺序
维护全局最低频次，除非新增数据，最低频次降为1，其他场景只会增加
如果访问最低频次数据，并且唯一，那么频次增加后此数据还是最低频次

❀1206. Design Skiplist

实现跳表

全量数组，部分数据作为索引
每个节点层数取决于随机抛硬币连续1次数，随机上升若干层
时间复杂度O(logn),空间复杂度O(n)即1+1/2+1/4+…
跳表相比较红黑树，实现上比较简单，范围查询效果更好

  • 每个节点是柱状节点，可以多层，设最大层号32即层号范围0-32
  • 高度取决于抛硬币连续1的次数，可以用随机数末尾0快速实现
  • 初始哨兵节点，表示负数最小值，跨满层

❀359. Logger Rate Limiter

10秒内没打印过就打印，否则不打印

可以维护滑动窗口，但是稍微比较复杂

简单的实现是维护每条信息上一次打印的时间，缺点是维护全量没有清理

933. Number of Recent Calls

每次在时间t收到电话，返回过去3000毫秒内的电话数

滑动窗口，题目保证时间升序用队列即可

应用设计

❀355. Design Twitter

实现增删关注，发推，获取最新10条内容

  • 推文维护
    • 集中链表维护，全局顺序完好，最新10条逻辑是1.取出关注列表2.扫描全局列表过滤出前10条。缺点是用户量可能很大，关注只是很小子集，扫描很多可能都轮不到
    • 分散维护，每个人自己维护链表。为了协调全局顺序，需要额外携带顺序标志。采用头插法，新数据在前
  • 获取最新
    • 动态维护（推模式），每个人预先维护长度10的内容链表，缺点是发推需要给所有关注者推送新内容，大V很多关注者开销巨大，并且增删关注也会影响
    • 实时计算（拉模式），从关注人中每人拿10条，进行归并，统计最新
  • Top策略
    • 所有关注人发推无脑全部置入优先队列，取10条
    • 先比较每个关注人头部数据，取最近更新的前10个关注人，每人10条，总数据量限制在100条，然后再筛选
  • 总策略
    • 每个人独立维护推文
    • 实时计算最新内容
    • 优化归并
