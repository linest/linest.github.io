Maven依赖管理

  • Maven
工具

依赖

如果一个项目包含所有功能的代码，那么无需任何依赖
然而现实中本着不重复造轮子的理念，项目往往需要依赖一些已有的库，也就是说要能运行项目，必须把已有的库引入过来
项目的依赖可以通过pom文件声明，而每一个依赖也有自己的pom,这样就构成了依赖网络图，需要注意的是不能有环形依赖

Maven具有传递依赖的特性，pom中声明了一个依赖，会根据依赖关系引入不止一个库

依赖版本管理

项目可能通过不同的依赖路径依赖了同一库的不同版本，这就是版本冲突
比如A->B->C->D2.0和A->E->D1.0

自动解决

默认行为是以路径短的为准。如果路径一样长，那么以定义时的顺序为准

手动指定

虽然A不直接引用D,但A可以在dependencyManagement中声明D的版本

依赖剔除

A可以引入E，同时不引入E的依赖D，在exclusion中声明

依赖存在范围

项目构建有很多阶段，不是所有的依赖都需要发布到最终产品中

  • complie：默认，编译时运行时都要的依赖
  • provided：只在编译时需要，运行时会由第三方提供，不会打进程序包
  • runtime: 只在运行时需要，编译时不需要
  • test: 只在测试时使用，不会打进程序包
  • system: 和provided类似，但必须显式指定依赖位置，不会去仓库查找
  • import: 只针对pom类型的依赖，引入dependencyManagement部分

依赖

独立的dependency标签表示真正存在依赖需要引入，而如果在dependencyManagement标签下，只表示管理依赖的策略，并不引入

通常在parent pom中定义dependencyManagement部分，声明每个依赖的版本，剔除部分，范围，这样具体的pom中只需要定义的groupId和artifactId即可
